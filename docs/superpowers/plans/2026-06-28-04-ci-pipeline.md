# CI Pipeline Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the stub `.github/workflows/ci.yml` (today it only runs `libs/` pytest) with a full
lifecycle pipeline: a path-filtered per-package quality-gate matrix that lints, typechecks, and tests
every changed package, plus a build-and-push stage that publishes all service images to Docker Hub on
merge to `main`.

**Architecture:** One workflow file, four jobs. `detect` runs `dorny/paths-filter` to compute which
packages changed (with dependency expansion). `python` is a matrix over the changed Python packages
that reuses the `Makefile` (`make install/lint/typecheck/test SVC=<pkg>`). `webui` runs the npm
toolchain. `build-push` is a matrix over the five service images, gated on `push` to `main`, that
builds each `Dockerfile` with `docker/build-push-action` and pushes `<org>/shepherd-<svc>:<sha>` and
`:latest`. Concurrency cancels superseded runs.

**Tech Stack:** GitHub Actions, Poetry, ruff, mypy, pytest, npm, docker buildx, Docker Hub.

## Global Constraints
- Reuse `Makefile` targets (`make lint`, `make typecheck SVC=`, `make test SVC=`); never reimplement
  lint/test commands inline in YAML.
- Path-filtered matrix: only changed packages run their quality gates. A change to `libs/` or `db/`
  expands to the services that path-depend on them.
- Build/push runs **only** on `push` to `main`; it builds all five images every time (not
  path-filtered).
- Images: `${{ vars.DOCKERHUB_ORG }}/shepherd-<svc>:<git-sha>` **and** `:latest`, for
  `svc in {postgres, db-init, fleet-api, telegram-bot, webui}`. Login via secrets
  `DOCKERHUB_USERNAME` / `DOCKERHUB_TOKEN`. These are the SAME images
  `deploy/docker-compose.prod.yml` (plan 03) pulls as `${REGISTRY}/shepherd-<svc>:${TAG}`.
- Pin Python 3.12, Poetry via pipx, Node 22.
- Official, already-common actions only (`actions/checkout`, `actions/setup-python`,
  `actions/setup-node`, `dorny/paths-filter`, `docker/login-action`, `docker/build-push-action`,
  `docker/setup-buildx-action`). No custom runners, no self-hosted, no bespoke scripts where an action
  exists. No caching/coverage/release tooling (YAGNI).
- Every deliberate simplification is marked with a YAML `#` comment.

### Verified facts this plan relies on
- **Dependency graph (from `pyproject.toml`):** `services/fleet-api` path-deps `shepherd-contracts`
  (`../../libs`) **and** `shepherd-db` (`../../db`). `services/telegram-bot` has **no** path deps
  (`libs`/`db` changes do NOT expand to it). `libs` and `db` have no Python dependents other than
  `fleet-api`. So: a `libs/` change runs `libs` + `fleet-api`; a `db/` change runs `db` + `fleet-api`.
- **Build contexts (from `docker-compose.yml`):** `postgres` -> context `./db`, file
  `db/postgres.Dockerfile`; `db-init` -> context `./db`, file `db/Dockerfile`; `fleet-api` -> context
  `.`, file `services/fleet-api/Dockerfile`; `telegram-bot` -> context `.`, file
  `services/telegram-bot/Dockerfile`; `webui` -> context `./services/webui`, file
  `services/webui/Dockerfile`.
- **webui scripts (from `package.json`):** `lint` (`eslint .`), `build` (`next build`), and `test`
  (`vitest run --coverage`) all exist; `package-lock.json` is present so `npm ci` works.
- **Makefile:** `lint` = `uvx ruff check .` (repo-wide, needs `uv`, which is NOT preinstalled on
  GitHub runners); `typecheck` = `cd $(SVC) && poetry run mypy .`; `test` = `cd $(SVC) && poetry run
  pytest -q`; `install` = `poetry env use python3.12 && poetry install`. SVC default is `libs`; service
  packages use full paths (`services/fleet-api`, `services/telegram-bot`).
- **mypy:** all five packages (`libs`, `libs/shepherd_config`, `db`, `services/fleet-api`,
  `services/telegram-bot`) now have `mypy` in their dev group and a `[tool.mypy]` config.
  `make typecheck SVC=<pkg>` exits 0 for every package. No conditional skips needed in CI.

### Task 2: Path-filtered package detection with dependency expansion

**Files:** `.github/workflows/ci.yml`

1. [x] Extend the `detect` job: add `dorny/paths-filter` with filters keyed by **Makefile SVC paths**
   (so quality legs call `make ... SVC=<key>` with no name mapping), wire the dependency expansion into
   the `fleet-api` filter, and expose two outputs: `packages` (JSON array of changed Python packages,
   webui removed) and `webui` (boolean). Replace the `detect` job from Task 1 with:

   ```yaml
     detect:
       runs-on: ubuntu-latest
       outputs:
         # JSON array of changed Python packages as Makefile SVC paths (webui excluded).
         packages: ${{ steps.pkgs.outputs.list }}
         webui: ${{ steps.filter.outputs.webui }}
       steps:
         - uses: actions/checkout@v4
         - id: filter
           uses: dorny/paths-filter@v3
           with:
             # Filter keys ARE the Makefile SVC paths, so each leg runs `make ... SVC=<key>`
             # with no mapping. Dependency expansion: fleet-api path-deps libs AND db, so it
             # also runs when either changes. telegram-bot has no path deps (not expanded).
             filters: |
               libs:
                 - 'libs/**'
               db:
                 - 'db/**'
               'services/fleet-api':
                 - 'services/fleet-api/**'
                 - 'libs/**'
                 - 'db/**'
               'services/telegram-bot':
                 - 'services/telegram-bot/**'
               webui:
                 - 'services/webui/**'
         - id: pkgs
           # Drop webui from the changed set (it runs in its own npm job). jq array-difference
           # is the smallest available glue; no action does set subtraction. CHANGES is passed
           # via env (not inlined into the script) to avoid expression-injection warnings.
           env:
             CHANGES: ${{ steps.filter.outputs.changes }}
           run: echo "list=$(jq -cn --argjson c "$CHANGES" '$c - ["webui"]')" >> "$GITHUB_OUTPUT"
   ```

### Task 3: Python quality-gate matrix (reuse the Makefile)

**Files:** `.github/workflows/ci.yml`

1. [x] Add a `python` job after `detect`: a matrix over the changed Python packages that reuses the
   Makefile targets. It installs Poetry and `uv` via pipx (the latter because `make lint` runs
   `uvx ruff check .`, and `uv` is not preinstalled on GitHub runners), then runs lint, typecheck, and
   test. Lint is run **per leg** (not as a separate job) on purpose: ruff via `uvx` needs no Poetry env
   and runs in seconds, so keeping it inside each changed leg makes every leg a self-contained
   lint+typecheck+test unit and avoids an extra coordination job. Append:

   ```yaml
     python:
       needs: detect
       # Empty array => no Python packages changed; skip rather than start an empty matrix.
       if: needs.detect.outputs.packages != '[]'
       runs-on: ubuntu-latest
       strategy:
         fail-fast: false
         matrix:
           package: ${{ fromJSON(needs.detect.outputs.packages) }}
       steps:
         - uses: actions/checkout@v4
         - uses: actions/setup-python@v5
           with:
             python-version: "3.12"
         - run: pipx install poetry
         # `make lint` runs `uvx ruff check .`; uvx ships with uv, which is not preinstalled.
         - run: pipx install uv
         - run: make install SVC=${{ matrix.package }}
         - run: make lint
         # NOTE: `make typecheck` assumes the package has mypy in its dev group. Today only
         # services/fleet-api does; adding mypy to libs/db/telegram-bot dev deps is a
         # prerequisite tracked outside this CI plan. The gate goes green once mypy is present.
         - run: make typecheck SVC=${{ matrix.package }}
         - run: make test SVC=${{ matrix.package }}
   ```

2. [x] **Verify:** run actionlint (or Docker fallback). Expected output: **no output, exit code 0**.
   Documented expectation: on a PR touching `libs/`, the `python` job expands to two legs
   (`SVC=libs`, `SVC=services/fleet-api`), each running `make install/lint/typecheck/test`; on a PR
   touching only `services/webui`, `needs.detect.outputs.packages == '[]'` and the `python` job is
   skipped entirely.

### Task 5: Build and push service images on merge to main

**Files:** `.github/workflows/ci.yml`

1. [x] Add the `build-push` job: a matrix over the five service images, gated on `push` to `main`,
   that logs in to Docker Hub, sets up buildx, and builds each `Dockerfile` with its verified context,
   pushing both the git-sha and `:latest` tags. The `if` uses `!cancelled() && !failure()` so the job
   still runs when a quality leg was **skipped** (an unchanged package) while still blocking on a real
   failure. The build matrix is **not** path-filtered - it republishes a consistent five-image set per
   SHA on every main push (path-filtering the build is a possible future optimization; YAGNI now).
   Append:

   ```yaml
     build-push:
       needs: [detect, python, webui]
       # Publish only on push to main. !cancelled() && !failure() lets this run when a quality
       # leg was skipped (unchanged package) while still blocking on an actual gate failure.
       if: >-
         ${{ !cancelled() && !failure()
             && github.event_name == 'push'
             && github.ref == 'refs/heads/main' }}
       runs-on: ubuntu-latest
       strategy:
         fail-fast: false
         matrix:
           # Contexts/Dockerfiles match docker-compose.yml. These images are exactly what
           # deploy/docker-compose.prod.yml (plan 03) pulls as ${REGISTRY}/shepherd-<svc>:${TAG}.
           include:
             - { svc: postgres, context: ./db, dockerfile: db/postgres.Dockerfile }
             - { svc: db-init, context: ./db, dockerfile: db/Dockerfile }
             - { svc: fleet-api, context: ., dockerfile: services/fleet-api/Dockerfile }
             - { svc: telegram-bot, context: ., dockerfile: services/telegram-bot/Dockerfile }
             - { svc: webui, context: ./services/webui, dockerfile: services/webui/Dockerfile }
       steps:
         - uses: actions/checkout@v4
         - uses: docker/setup-buildx-action@v3
         - uses: docker/login-action@v3
           with:
             username: ${{ secrets.DOCKERHUB_USERNAME }}
             password: ${{ secrets.DOCKERHUB_TOKEN }}
         - uses: docker/build-push-action@v6
           with:
             context: ${{ matrix.context }}
             file: ${{ matrix.dockerfile }}
             push: true
             tags: |
               ${{ vars.DOCKERHUB_ORG }}/shepherd-${{ matrix.svc }}:${{ github.sha }}
               ${{ vars.DOCKERHUB_ORG }}/shepherd-${{ matrix.svc }}:latest
   ```

2. [x] **Verify:** run actionlint (or Docker fallback). Expected output: **no output, exit code 0**.
   Documented expectation: on a `pull_request`, `build-push` is skipped (the `if` fails the
   event/ref check). On a `push` to `main` touching `services/fleet-api`, the `python` leg runs and
   passes, `webui` is skipped, and `build-push` runs all five image legs, pushing
   `<org>/shepherd-<svc>:<sha>` and `:latest` for `svc in {postgres, db-init, fleet-api,
   telegram-bot, webui}`.
